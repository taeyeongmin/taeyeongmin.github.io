사내 그룹웨어 유지보수

전자결재·메일·PMS 등 사내 업무 흐름을 지탱하는 그룹웨어를 운영하면서 체감 성능과 데이터 안정성을 동시에 개선했습니다. 운영 병목의 제거와 품질 개선, PMS 메뉴 추가에 초점을 둔 프로젝트입니다.

핵심 개선 포인트

  • 메일 발송 처리 병목 제거(응답 지연 개선)
  • 인덱스 재정비 및 실행 계획 기반 쿼리 개선
  • 전자결재 트리 구조 뷰 가독성/탐색성 개선
  • PMS 메뉴 추가를 위한 테이블 정합성 검토

내 역할

  • 개발 PL역할 수행
  • 기획,디자인팀과의 커뮤니케이션
  • 운영 이슈 재현/원인 분석(로그·쿼리·실행계획 중심)
  • 개선안 도출
  • 배포 전후 검증 기준 수립

1. 서비스 개요 | 배경과 목적

그룹웨어는 임직원이 매일 사용하는 업무 기반 시스템이라, 작은 지연도 반복되면 “업무 체감”으로 크게 누적됩니다. 유지보수 기간 동안 사용자 문의/운영 이슈가 빈번했던 구간을 중심으로 성능·정합성·가독성 관점에서 개선 우선순위를 잡았습니다.

운영 환경에서 반복적으로 발생하던 문제

현상 중심 정리
  • 메일 발송이 요청 처리 흐름을 막아 응답 지연이 발생
  • 인덱스/쿼리 구조가 현재 트래픽 패턴과 맞지 않아 병목이 누적
  • 전자결재 라인/결재 흐름을 파악하기 어려운 화면 구조

개선 목표

운영 안정성
  • 사용자 요청 흐름에서 즉시 개선 가능한 지연 제거
  • PMS 메뉴 추가 및 타 메뉴와의 데이터 연동
  • 실행 계획 기반으로 지속되는 병목부터 제거
  • 확장 작업에서도 정합성을 유지할 수 있는 기준 확보

2. 진행한 일 | 문제 해결

프로젝트 리딩 및 운영 조율

PL 역할
팀 구성

총 4명의 개발 인원(본인 포함)으로 구성된 팀에서 개발 PL 역할을 맡아 업무 분배와 일정 관리, 코드 품질 기준을 조율했습니다.

조율 및 커뮤니케이션

기획/운영 부서와의 요구사항 정리, 변경 범위 산정, 배포 일정 조율을 담당하며 개발과 타 부서 간의 완충 역할을 수행했습니다.

형상 및 일정 관리

브랜치 전략과 머지 기준을 정리하고 작업 단위별 일정 산정을 통해 병목 구간을 사전에 조정했습니다.

유지보수 프로젝트 특성상 안정성이 중요했기 때문에 작업 단위를 작게 쪼개고 변경 범위를 명확히 정의한 뒤 배포했습니다. 그 결과 개선 작업 중 서비스 중단이나 주요 장애 없이 점진적 품질 향상이 가능했습니다.

메일 발송 처리 개선 (동기 → 비동기 + 사전 실패 대응 설계)

성능/운영 안정성
배경

결재/요청 처리 시 Naver Works 메일 API를 호출하는 로직이 요청 흐름에 포함되어 있었고 사용자는 완료 버튼 이후 3~4초의 응답 지연을 체감하고 있었습니다.

1차 개선

메일 발송 로직에 @Async를 적용하여 사용자 요청과 외부 메일 API 호출을 분리했습니다. 그 결과 응답 시간은 3~4초에서 0.5초 수준으로 단축되었습니다.

고려한 리스크

비동기 전환 이후 외부 API 호출 실패가 발생할 경우 예외가 사용자 요청과 분리되어 즉시 인지되지 않을 수 있다는 점을 염두에 두었습니다. 또한 네트워크 지연이나 일시적 API 오류가 발생했을 때 단발성 호출로 끝나면 메일 누락 가능성이 존재한다고 판단했습니다.

  • 비동기 예외가 사용자 요청과 분리되어 실패가 드러나지 않을 가능성
  • 외부 API 일시 오류 발생 시 재시도 없이 종료되는 구조적 한계
  • 요청 성공과 메일 발송 성공이 동일하게 인식되는 설계 위험
사전 보완 조치
실제 장애가 발생한 것은 아니었지만 운영 환경에서 발생 가능한 리스크를 고려해 비동기 예외를 기록하도록 보완했습니다. 또한 외부 API 호출 실패 시 일정 횟수 재시도 로직을 추가하고 반복 실패 건은 별도로 확인 가능하도록 분리해 운영에서 추적 가능한 구조로 개선했습니다.

이를 통해 응답 성능 개선과 함께 외부 API 의존 구간에서의 잠재적 실패 리스크를 사전에 낮출 수 있었습니다. 비동기 처리는 성능 최적화 수단이지만 실패 처리 전략까지 함께 설계해야 운영 품질이 완성된다는 점을 체감했습니다.

인덱스 재정비 및 쿼리 성능 개선

DB/운영 품질
배경

업무 목록/검색 계열 화면은 조회 트래픽이 꾸준히 발생했고 작은 비효율이 전체 체감 성능으로 연결됐습니다.

문제

필요 이상의 인덱스가 존재하거나 실제 조건절과 맞지 않는 인덱스가 사용돼 정렬/필터 구간에서 지연이 발생했습니다.

해결

실행 계획을 기준으로 사용되지 않는 인덱스를 정리하고 자주 쓰는 조건/정렬 패턴에 맞춘 인덱스를 새로 설계했습니다.

인덱스 변경은 운영 리스크가 크기 때문에 영향 범위를 좁히기 위해 대상 화면과 쿼리 패턴을 먼저 고정한 뒤 변경 전후 실행 계획을 비교하고 배포 범위를 단계적으로 가져갔습니다.

전자결재 트리 구조 뷰 개선

UX/업무 흐름
배경

결재는 조직 커뮤니케이션의 핵심이라 “누가 어떤 순서로 승인해야 하는지”가 한눈에 보여야 했습니다.

문제

기존 화면은 결재 라인의 구조가 평면적으로 보였고 결재 흐름을 파악하기 위해 스크롤과 클릭이 반복됐습니다.

해결

계층 관계가 드러나는 트리 형태로 정보를 재구성해 결재 흐름을 화면에서 빠르게 파악할 수 있도록 개선했습니다.

PMS 메뉴 확장 및 기존 시스템 연동

대규모 기능 확장
배경

PMS 메뉴는 기능 규모가 큰 영역이었으며 단순 화면 추가가 아닌 기존 그룹웨어 메뉴들과의 연동이 필요한 확장 작업이었습니다. 조직/업무 단위별 사용 흐름에 깊이 연결되어 있는 구조였습니다.

핵심 과제

신규 PMS 기능을 추가하면서도 기존 프로젝트 관련 메뉴들과 연결되도록 설계해야 했습니다. 특히 상태 값, 데이터 참조 흐름이 충돌하지 않도록 구조를 정리하는 것이 중요했습니다.

해결

기존 메뉴들과의 연계 지점을 먼저 도식화하고 데이터 흐름과 참조 관계를 정리한 뒤 확장 구조를 설계했습니다. 변경 범위를 단계적으로 적용하여 기존 기능에 영향 없이 통합을 완료했습니다.

단순 기능 추가가 아니라 기존 시스템과 유기적으로 연결되는 확장 작업이었기 때문에 데이터 정합성과 권한 체계를 함께 고려하며 통합 구조를 설계했습니다. 그 결과 기존 메뉴에 대한 회귀 이슈 없이 PMS 기능을 안정적으로 확장할 수 있었습니다.

3. 성장한 점 | 성과와 배운 점

유지보수 프로젝트에서 가장 중요한 것은 변경의 영향 범위를 통제하면서도 사용자 체감을 개선하는 것이라고 느꼈습니다. 이번 경험을 통해 운영 환경에서의 개선 작업을 더 안전하고 일관되게 진행할 수 있게 됐습니다.

성과

정량/정성
  • 응답 지연의 주요 원인을 분리해 사용자 체감 성능을 개선
  • 실행 계획 기반으로 인덱스/쿼리 병목을 정리하는 기준 확보
  • 전자결재 화면 구조 개선으로 정보 탐색 비용을 낮춤

배운 점

운영 관점
  • 비동기 전환은 성능뿐 아니라 실패 처리/재시도/모니터링까지 포함해야 완성된 개선이 된다
  • 인덱스 변경은 기술적 정답보다 배포 전략과 검증 기준이 더 중요할 수 있다
  • 화면 개선도 결국 운영 비용(문의/업무 지연)을 줄이는 개발이라는 관점이 필요하다
