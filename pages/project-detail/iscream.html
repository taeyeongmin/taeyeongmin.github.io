Work Project · 이커머스 재구축

아이스크림 몰 사이트 재구축

플래티어(이커머스 솔루션) 기반의 MSA 환경에서 고객사 요구사항에 맞춘 기능 확장·정책 정비를 수행했습니다. 저는 장바구니·주문·배송 API를 담당하여 구매 플로우의 안정성과 확장성을 강화하는 데 집중했습니다.

핵심 구현 포인트

  • 상품 타입 확장(추가 입력/추가 구성) 정책을 장바구니·주문 구조에 반영
  • 주문 변경 가능 범위/조건(단품 옵션, 동일 가격 등) 기준 정리 및 API 반영
  • 합배송(아이스크림 물류센터 경유) 프로세스에 맞춘 배송 상태/흐름 수정
  • 장바구니 성능 저하 구간 분석 후 병목 제거(조회/저장 흐름 개선)

내 역할

  • 장바구니 도메인 API 개발
  • 주문 도메인 API 개발(FO,BO)
  • 기존(플래티어 제공) 주문 전략 패턴 구조를 이해하고 장바구니에 확장 적용
  • 정책 문서 기반으로 예외 케이스(옵션/가격/상태) 표준화 및 검증 로직 반영
  • 타 도메인(상품/혜택/정산/입점사 등)과 연동 포인트 정리 및 협업

1. 서비스 개요 | 배경과 구조

본 프로젝트는 처음부터 모든 것을 새로 만드는 구축이라기보다 플래티어에서 제공하는 이커머스 기본 기능(프레임/정책/확장 포인트)을 기반으로 고객사의 요구사항에 맞춰 정책을 재정의하고 기능을 수정/확장하는 형태였습니다. 또한 서비스가 MSA로 분리되어 있어 장애/이슈 발생 시 단일 서버처럼 한 번에 추적하기 어렵다는 특성이 있었습니다.

프로젝트 특성

솔루션 기반 · MSA
  • 프로젝트의 골격과 DB 스키마는 솔루션이 제공하며 새로운 요구에 맞춰 커스터마이징 및 추가
  • 객체지향 특성을 극대화환 구조 및 설계
  • 한 번의 로직 흐름이 여러 서비스/DB를 거치므로 트러블슈팅 난이도가 높음

나의 구현 목표

  • 주문 → 재고 차감 → 포인트 적립 → .... → 결제 → 후처리까지 이어지는 전체 흐름이 정합성과 일관성을 유지하도록 구현
  • 장바구니/주문 구조를 “상품 타입 확장”에 견딜 수 있게 확장
  • 주문 변경/배송 상태 흐름을 정책 문서 기준으로 표준화
  • 운영 이슈 대응을 위해 “원인 좁히기”가 가능한 단서(식별자/로그)를 강화

2. 진행한 일 | 문제 해결

솔루션 기반 커스터마이징: 기존 구조를 이해하고 안전하게 확장하기

배경

플래티어 솔루션은 이커머스 핵심 기능과 확장 포인트(패턴/정책)가 이미 구현되어 있어 새로운 구현보다 기존 것들의 확장이 더 중요했습니다.

접근

기존 주문 영역의 전략 패턴 구조(전략 선택 → 컨텍스트 구성 → 실행 흐름)를 먼저 분석한 뒤 장바구니 담기/수정에도 동일한 방식으로 확장해 일관성을 유지했습니다.

효과

고객사 요구사항이 늘어도 전략 추가/교체 중심으로 확장 가능해졌고 기존 기능(주문)의 설계 철학과도 어긋나지 않게 구현할 수 있었습니다.

포인트
주문 전략 패턴 자체는 솔루션에 이미 존재했고 저는 장바구니 담기/수정을 해당 구조에 맞춰 확장했습니다. (기존 구조를 참조해, 동일한 선택/실행 흐름으로 구현)

상품 타입 확장 대응: 추가 입력/추가 구성 장바구니 정책 반영

배경

단순 옵션(색상/사이즈)을 넘어 문구 입력 등 추가 입력 옵션과 세트/부속품 형태의 추가 구성 옵션 요구가 발생했습니다.

문제

기존 장바구니 구조에서는 본품+옵션 중심으로만 가정하고 있어 추가 입력/추가 구성의 행 구성/수량 변경/삭제 정책을 명확히 처리하기 어려웠습니다.

해결

정책을 먼저 확정한 뒤(행 적재 규칙, 단독 주문 불가 등) 장바구니 담기/수정/삭제 API와 저장 구조를 해당 규칙에 맞춰 재정비했습니다.

  • 추가 입력 옵션: 존재 시 장바구니에 무조건 별도 행으로 적재
  • 추가 구성 옵션: 개별 삭제/수량 변경 가능하되 본품 삭제 시 함께 제거
  • 추가 구성 단독 담기/주문 불가: 본품과의 종속 관계를 저장 구조/검증 로직에 반영
운영 관점 보완
정책이 먼저 구현이 나중 순서로 접근해 예외 케이스(부분 삭제, 수량 변경, 본품 삭제 동반 처리)에서 운영 문의가 발생하지 않도록 행 단위 규칙을 명확히 정의하고 API 응답도 일관되게 맞췄습니다.

주문 정책 표준화: 주문 변경 가능 범위/조건을 API로 고정

배경

운영 과정에서 주문 변경은 장애/클레임으로 이어질 수 있어 가능한 범위와 조건을 시스템적으로 제한하는 것이 중요했습니다.

문제

변경 가능한 단위(상품/옵션)와 가격 조건이 애매하면 정산/재고/혜택 적용에서 사이드 이펙트가 커질 수 있었습니다.

해결

정책 문서 기준으로 변경 허용 범위를 제한하고 서버 검증 로직으로 불가능한 변경을 조기에 차단했습니다.

  • 주문 변경(상품)은 상품 내 단품 상품(옵션) 단위에서만 가능
  • 가격이 같을 때만 변경 가능 (정산/혜택/쿠폰 영향 최소화)
  • 추가 입력 옵션 값은 주문 테이블에 반영 가능한 형태로 확장

이 작업은 기능 추가라기보다 운영 중 발생 가능한 예외를 규칙으로 못 박아 장애성 이슈를 줄이는 작업에 가까웠습니다. 무엇이 가능한지보다 무엇을 불가능하게 만들지를 먼저 결정한 경험이었습니다.

FO · BO 주문 API 통합 설계: 구매부터 운영 처리까지 일관된 상태 흐름 확립

주문은 단순한 데이터 생성이 아니라 고객(FO)의 구매 행위와 운영자(BO)의 후처리 행위가 연결된 하나의 라이프사이클입니다. 저는 FO 주문 API와 BO 주문 API를 각각 구현하면서도 두 영역이 동일한 상태 규칙과 정책 위에서 동작하도록 설계하는 데 집중했습니다.

FO 주문 API (고객 영역)

장바구니 기반 주문 생성 주문 상세 조회 주문 변경 API를 담당했습니다. 주문 생성 시 상품/옵션/추가입력/추가구성 데이터를 서버 기준으로 재검증하고 허용된 변경 범위만 수행되도록 정책을 고정했습니다.

BO 주문 API (관리자 영역)

주문 목록/상세 조회, 주문 상태 변경, 배송 지시/출고 처리 API를 구현했습니다. BO 화면의 버튼 노출 정책과 서버 상태 전이 조건을 동일 기준으로 맞춰 화면-서버 정책 불일치를 제거했습니다.

공통 설계 포인트

주문 상태를 단순 Enum 변경이 아닌 업무 규칙 기반 상태 전이로 관리했습니다. 허용되지 않은 상태 변경 요청은 서버에서 사전 차단해 운영 실수로 인한 데이터 왜곡 가능성을 줄였습니다.

  • 주문 생성 단계 분리: 검증 → 주문 엔티티 구성 → 상태 확정 → 연동 준비
  • 동일 가격/옵션 단위 변경 등 정책을 서버 규칙으로 고정
  • 주문번호/장바구니키 기준 로그 정리로 MSA 환경에서 추적성 강화
  • FO와 BO가 동일 상태 흐름을 공유하도록 상태 전이 구조 통합
FO와 BO를 분리된 API가 아닌 하나의 주문 라이프사이클 관점으로 설계함으로써
- 주문 생성 이후 상태 혼선 감소
- BO 처리 과정에서의 예외 케이스 축소
- 전략 패턴의 이점을 체감, 주문의 전략이 잘 구성되어 있어 수정 시 용이하였음
- 정책 변경 시 영향 범위를 명확히 파악 가능한 설계 구조 확보

장바구니 성능 저하 구간 분석 및 병목 제거

배경

장바구니는 구매 플로우 초입에서 반복 호출되며 작은 지연도 사용자 체감에 직접적으로 누적되는 영역이였습니다.

문제

상품/옵션/추가구성까지 조합되며 조회·계산 범위가 커져 특정 상황에서 응답이 느려지는 구간이 확인되었습니다.

해결

어떤 조합에서 느려지는지를 케이스로 분리한 뒤 불필요한 중복 조회/계산을 줄이고 저장/조회 흐름을 정리했습니다.

특히 추가 입력/추가 구성 같은 확장 정책은 기능은 맞는데 느린 상태가 되기 쉬워 정책을 반영한 뒤에도 실제 조회 패턴에서 병목이 생기지 않도록 점검하고 개선했습니다.

3. 성장한 점 | 성과와 배운 점

이번 프로젝트는 기능을 구현했다를 넘어 이커머스 도메인의 이해와 객체 지향 설계의 이점,활용법을 체득한 경험이었습니다.

성과

  • 정확성과 일관성이 보장된 주문 기능 구현
  • 추가 입력/추가 구성 등 상품 타입 확장 요구를 장바구니 규칙으로 안정화
  • 주문 변경 가능 조건을 서버 규칙으로 고정하여 운영 리스크를 감소
  • 장바구니 성능 병목을 케이스 단위로 분해하여 체감 지연 구간 개선

배운 점

  • 솔루션 기반 개발은 새로 만드는 능력보다 기존 설계를 읽고 확장하는 능력이 중요하다
  • 이커머스 핵심은 기능 자체보다 정책/예외/상태를 시스템 규칙으로 고정하는 데 있다
  • DB index 활용 능력과 db 실행계획 분석 능력 향상
  • 디자인 패턴은 멋내기가 아니라 변경을 추가/교체로 바꾸는 운영 친화적 도구다
